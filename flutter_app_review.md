# Flutter App 專項 Review（Android）

日期：2026-03-07  
範圍：`mobile_app/`（以目前程式實作為準，文件作為參考）  
評估嚴格度：中度嚴格（兼顧短期可落地與中期可產品化）

---

## 一、Review 著重原則與目標

### 著重原則
1. **以使用者實際體驗與可上線風險優先**：先處理會造成資料遺失、上傳失敗、錯誤不可觀測的問題。
2. **以程式實作為主、規劃文件為輔**：設計文件若與實作衝突，先記錄為「文件落差」，不直接當成實作缺陷。
3. **以端到端流程為主線**：登入 → 相簿選擇 → 掃描 → 背景上傳 → 狀態追蹤 → 驗證完成。
4. **建議必須可執行**：每一項問題都附上影響、優先級（P0/P1/P2）與修正方向。

### 本次目標
- 確認專案目前是否達到 MVP「可穩定備份」門檻。
- 找出 UI/UX 的清晰度與優雅度缺口，提出簡潔優化路徑。
- 評估 Flutter Android 架構是否可持續擴充、可測試、可維運。
- 盤點程式碼品質、邏輯正確性與測試覆蓋缺口。

---

## 二、依主要組件切分的 Review 項目

### 1) 前端（Flutter UI）
- 畫面流程一致性：登入、掃描、上傳、設定、錯誤回饋。
- 互動可理解性：按鈕文案、狀態標示、失敗後可恢復性。
- 視覺一致性：字級、間距、顏色語意、元件密度。
- 效能體感：列表刷新、輪詢更新、相簿縮圖載入。

### 2) 後端/API（Google Photos 串接）
- OAuth Token 取得與續期安全性。
- 相簿建立與快取策略是否可避免重複建立。
- 上傳協定是否真正落地 resumable/chunked。
- 失敗重試與 Rate Limit（429）因應策略。

### 3) App 架構
- 模組邊界（auth/albums/upload/database/settings）是否清晰。
- 狀態管理是否一致（Riverpod vs setState 混用影響）。
- 背景任務與資料庫狀態轉移是否可驗證。
- 可觀測性（錯誤紀錄、指標、追蹤）是否足以營運。

### 4) 測試與品質
- 單元測試覆蓋核心風險（上傳、權限、token、重試、驗證）。
- UI/整合測試是否覆蓋主要使用者路徑。
- CI 是否涵蓋 Flutter build/test/lint。
- 靜態檢查規範是否足夠（lint 強度、例外處理規範）。

## 三、角色 1：PM 觀點（目標功能符合度）

### 結論
目前功能已達 **可操作 MVP**，但尚未達「可放心長時間背景備份」的產品穩定門檻。  
主要風險在於：上傳協定落地完整性、驗證流程未接線、失敗恢復策略不足。

### 功能符合現況（以實作為準）
- ✅ 已有 Google 登入與靜默登入流程。
- ✅ 已有相簿選取、掃描、Hash 防重複、本地狀態追蹤。
- ✅ 已有背景上傳排程與狀態儀表板。
- ✅ 已有 Wi‑Fi only 設定、錯誤日誌檢視、重置卡住任務。
- ⚠️ 「上傳後驗證」實作存在但未接入主流程（`VerificationWorker` 未被調用）。
- ⚠️ 設計文件宣稱 Isar/WorkManager，實作為 sqflite/background_downloader，文件與程式不一致。

### PM 風險清單與優先級
1. **P0：上傳完成定義不夠嚴格**  
   - 影響：使用者看到「完成」但遠端可能尚未最終可見或未驗證。  
   - 建議：將 `done` 與 `verified` 明確區分在 UI；加入驗證工作排程與重試。

2. **P0：Resumable/chunked 與實際上傳路徑有斷層**  
   - 影響：大型檔案/網路波動下穩定性不可預期。  
   - 建議：統一單一路徑（要嘛真正 chunked resumable，要嘛完整證明 downloader 路徑可恢復）。

3. **P1：文件與實作落差大，影響團隊協作與交接**  
   - 影響：新人或 QA 會依錯誤心智模型測試。  
   - 建議：更新 mobile README 與設計文件，標注「目前實作版架構」。

4. **P1：缺少 Flutter CI，回歸風險偏高**  
   - 影響：改動容易破壞主流程且無自動防線。  
   - 建議：新增 mobile lint/test/build 的 workflow。

### PM 驗收目標建議（下一里程碑）
- 真機連續上傳 5,000+ 媒體不中斷或可恢復。
- 網路切換（Wi‑Fi/4G）下狀態一致且可追蹤。
- 驗證成功率與失敗重試策略可量化（儀表數據可看）。

## 四、角色 2：Android UI/UX 設計師觀點

### 結論
整體風格已朝簡潔方向前進，核心流程清楚；但「視覺系統一致性」與「狀態語意清晰度」仍不足，尚未達到「優雅且高可信任」體驗。

### 目前優點
- 登入頁有明確主 CTA、資訊密度適中。
- 相簿選取頁縮圖卡片視覺良好，選取狀態辨識清楚。
- Dashboard 能呈現相簿層級進度，對「備份」任務很關鍵。

### 主要 UX 問題
1. **狀態語意不夠一致（P0）**
   - `done` 與 `verified` 對一般使用者難以理解，UI 文案未區分「已提交」與「已確認存在」。

2. **錯誤回饋偏工程向（P1）**
   - 多處直接顯示 exception 字串，對非技術使用者不友善。

3. **資訊分層可再簡化（P1）**
   - 設定、維運工具、危險操作同層放在 Drawer，缺乏操作風險分組。

4. **一致性不足（P2）**
   - 顏色、字體粗細、陰影與圓角雖有主題，但仍有多處硬編碼，後續擴充會破壞統一感。

### UI/UX 優化建議（保持簡潔，不擴增複雜功能）
1. **建立三段式備份狀態文案（P0）**
   - 建議：`等待上傳` / `已上傳待驗證` / `已驗證完成`。
   - 在 Dashboard 以簡短 legend 呈現，降低誤解。

2. **錯誤文案分級（P1）**
   - 使用者可懂版：例如「網路不穩，稍後自動重試」。
   - 技術詳情收在「錯誤日誌」內，不直接丟例外訊息。

3. **操作風險分區（P1）**
   - Drawer 內將「一般設定」與「維運/危險操作（重置、清庫）」視覺分段。

4. **收斂設計 token（P2）**
   - 以 Theme/常數統一色彩與字級，移除散落硬編碼值。

## 五、角色 3：架構師觀點（架構與技術正確性）

### 結論
目前採「feature 模組化 + Riverpod + 本地 DB + 背景任務」方向是正確的；但在資料一致性與背景上傳策略上仍有關鍵斷點，需先補齊才能稱得上穩健架構。

### 架構正向點
- 功能切分清楚：`auth`、`albums`、`upload`、`database`、`settings`。
- Repository 與 Service 分層已有雛形。
- `GooglePhotosApiClient` 具 lock 機制，能降低同時建相簿/批次呼叫競態。

### 關鍵架構議題
1. **上傳路徑雙軌但未統一（P0）**
   - 有 `ResumableUploadEngine.uploadFileInChunks()`，但主流程使用 downloader 上傳 finalize。
   - 風險：行為定義分裂、故障排查困難。

2. **驗證流程存在但未編排（P0）**
   - `VerificationWorker` 未串入排程/事件；目前有「邏輯存在但系統不會跑」問題。

3. **狀態同步採 polling（P1）**
   - Dashboard 每 2 秒輪詢 DB，簡單可行但耗電與效能可預期上升。

4. **資料模型未涵蓋驗證關鍵欄位（P1）**
   - 缺少可穩定驗證所需欄位（例如 remoteAlbumId/verifyAttempt 等），導致驗證策略難精準落地。

5. **設定/環境管理尚未產品化（P1）**
   - `serverClientId` 直接寫在程式中，缺乏 build flavor 或環境注入策略。

### 架構修正方向（建議順序）
1. **先統一上傳協定實作（P0）**：定義唯一真實來源流程與故障恢復機制。
2. **接線驗證工作流（P0）**：上傳成功後排入驗證，驗證失敗可回退或重試。
3. **明確狀態機（P1）**：定義合法轉移 `pending→uploading→done→verified/failed`。
4. **事件驅動取代高頻輪詢（P2）**：減少不必要 DB polling。

## 六、角色 4：專業 Flutter 工程師觀點（程式碼品質與邏輯）

### 結論
程式碼可讀性中上、命名清楚，具備基礎測試；但錯誤處理、可測試性與可靠性控制仍需補強，否則遇到真實大量媒體場景時風險偏高。

### 程式品質觀察
1. **優點**
   - 主要流程可追蹤，功能拆分合理。
   - DB repository 與 Hash service 有單元測試。
   - 有基本使用者恢復工具（重置 stuck task、清除資料庫）。

2. **待改善**
   - **P0**：多處以 `print` + 字串例外處理，缺乏統一錯誤型別與錯誤碼。
   - **P1**：`UploadManager` 責任偏大（排程、回呼、API 寫入、錯誤紀錄都在同類別）。
   - **P1**：Repository 直接暴露 SQL 操作給 UI 周邊流程（部分邏輯在畫面層執行 raw SQL）。
   - **P1**：部分 async 流程 fire-and-forget（`_processAndEnqueue`）不易做結果保證。
   - **P2**：UI/Theme 常數分散、硬編碼偏多。

### 測試覆蓋現況
- 已有：model/hash/settings/repository 的 unit tests。
- 缺少：
  - UploadManager 與 API client 的錯誤路徑測試（429、401、timeout、network flip）。
  - Widget tests（登入、相簿選擇、Dashboard 狀態顯示）。
  - 整合測試（完整掃描→上傳→狀態轉移）。

### 工程落地建議
1. **P0：先補高風險測試**（上傳狀態轉移與重試策略）。
2. **P1：拆分 UploadManager**（Task 排程、狀態更新、API 後處理分離）。
3. **P1：建立 domain error model**（可在 UI 顯示友善訊息）。
4. **P1：新增 Flutter CI**（`flutter analyze` + `flutter test` + Android build smoke）。
5. **P2：重構主題常數與字串管理**（為 i18n 與視覺一致性鋪路）。

## 七、綜合優先級 Roadmap（建議）

### P0（本週）
1. 統一上傳主流程（resumable 策略定版並落地）。
2. 串接驗證工作流，UI 區分 `done` 與 `verified`。
3. 補上傳核心失敗情境測試（token 失效、429、網路中斷）。

### P1（2–4 週）
1. 新增 mobile CI pipeline（analyze/test/build）。
2. 拆分 UploadManager 並標準化錯誤模型。
3. 更新文件（mobile README + 架構圖 + 狀態機說明）。

### P2（中期）
1. 優化 Dashboard 更新機制（降頻輪詢或事件驅動）。
2. UI token 化與字串/i18n 規劃。
3. 可觀測性提升（結構化 log、上傳成功率指標）。

## 八、最終評語

這個 Flutter App 已具備「可跑的備份 MVP」與不錯的功能骨架，但尚未達到高信賴備份工具應有的穩定度。  
若依本 review 的 P0/P1 順序補齊，短期可明顯提升成功率與可維護性；中期可進入可持續迭代的產品化節奏。
